## Preparation

Firstly, let's enhance the clarity of our code by renaming the default get("/") to the more specific get("/quote") in both ApplicationTest.kt and Routing.kt. This not only makes our route more explicit but also sets the stage for a cleaner separation of routing logic. To achieve this, we'll create a separate file named SimpleStockQuoteRoute.kt, where we'll house the routing logic. This restructuring lays the groundwork for seamlessly incorporating new endpoints in the future.

```kotlin
fun Route.simpleStockQuoteRoute(
    apiKey: String?,
    alphaVantageApi: AlphaVantageApi,
) {
    get("/quote") {
        apiKey?.let {
            val quote = alphaVantageApi.getQuote(
                symbol = "IBM",
                apiKey = apiKey,
            )
            call.respond(quote)
        } ?: call.respond("Failed to get the API key!")
    }
}
```

With this, we can update Routing.kt to leverage the new function in SimpleStockQuoteRoute.kt.

```kotlin
fun Application.configureRouting(...) {
  //...
  routing {
    simpleStockQuoteRoute(apiKey, alphaVantageApi)
  }
}
```

## Path Parameters

In our current implementation, we've hard-coded symbol = "IBM" to consistently retrieve stock information for IBM, which isn't equivalent to wrapping the original AlphaVantage endpoint. Fortunately, Ktor makes it straightforward to support path parameters.

```kotlin
// A path parameter ({param}) matches a path segment and captures it
// as a parameter named param. In this case, /quote/{symbol}
// matches /quote/IBM, but doesn't match /quote.
get("/quote/{$SYMBOL_PARAM}") {
    // To access the parameter, use the call.parameters as such
    val tickerSymbol = call.parameters[SYMBOL_PARAM]
    if (apiKey != null && tickerSymbol != null) {
        val quote = alphaVantageApi.getQuote(
            symbol = tickerSymbol,
            apiKey = apiKey,
        )
        call.respond(quote)
    } else {
        call.respond("Failed to get the API key!")
    }
}

private const val SYMBOL_PARAM = "symbol"
```

Now, we have a dynamic endpoint that can fetch stock information for any public company supported by the AlphaVantage endpoint. However, providing a simple text response when an error occurs doesn't offer the best user experience. In the upcoming section, we'll address this and enhance the error-handling experience.

## Error Handling

To illustrate error handling, let's introduce a simple data class to encapsulate error messages.

```kotlin
@Serializable
data class ErrorMessage(
    val errorMessage: String,
)
```

Next, let's modify the logic for displaying error messages in simpleStockQuoteRoute. Currently, the endpoint displays "Failed to get the API key!" even when the ticker symbol (not the API key) is null.

```kotlin
get("/quote/{$SYMBOL_PARAM}") {
    apiKey?.let {
        tickerSymbol?.let {
            val quote = alphaVantageApi.getQuote(
                symbol = tickerSymbol,
                apiKey = apiKey,
            )
            call.respond(quote)
        } ?: call.respond(
            ErrorMessage(errorMessage = "Failed to get the ticker symbol!")
        )
    } ?: call.respond(
        ErrorMessage(errorMessage = "Failed to get the API key!")
    )
}
```

When an unrecognized ticker symbol is passed, the AlphaVantage endpoint returns an empty object, leading to a peculiar message. To address this, we can allow nullable fields when parsing the AlphaVantage response.

```kotlin
@Serializable
data class GlobalQuote(
    @SerialName("01. symbol")
    val symbol: String? = null,
    @SerialName("02. open")
    val open: Double? = null,
    @SerialName("03. high")
    val high: Double? = null,
    @SerialName("04. low")
    val low: Double? = null,
    @SerialName("05. price")
    val price: Double? = null,
    @SerialName("06. volume")
    val volume: Long? = null,
    @SerialName("07. latest trading day")
    val latestTradingDay: String? = null,
    @SerialName("08. previous close")
    val previousClose: Double? = null,
    @SerialName("09. change")
    val change: Double? = null,
    @SerialName("10. change percent")
    val changePercent: String? = null,
)
```

Consequently, fields in SimpleStockQuote should be nullable as well.

```kotlin
@Serializable
data class SimpleStockQuote(
    val price: Double?,
    val symbol: String?,
    val change: Double?,
    val changePercent: String?,
)
```

Now, when calling the endpoint with an unrecognized ticker symbol, the response simply displays null fields.

```json
{
    "price": null,
    "symbol": null,
    "change": null,
    "changePercent": null
}
```

From here, you can decide whether this level of information is sufficient or check the fields to see if they're all null, providing a clearer error message if needed.
